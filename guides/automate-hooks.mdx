---
title: "Create GatomIA Hooks to automate spec and agent workflows"
description: "Create GatomIA Hooks that fire MCP tool calls, Git operations, GitHub API actions, or custom agents automatically when spec and agent operations complete."
---

GatomIA Hooks are automation rules that connect agent operations to downstream actions. When a trigger fires — for example, after the SpecKit `plan` operation completes — GatomIA executes the configured action automatically. You can use hooks to create GitHub issues, commit generated files, run a code review agent, or call any MCP tool, all without manual intervention.

## Create your first hook

<Steps>
  <Step title="Open the Hooks view">
    Click the GatomIA icon in the Activity Bar and select the **Hooks** view. An empty state with an **Add Hook** button appears if no hooks exist yet.
  </Step>
  <Step title="Click Add Hook">
    Click the **+** (Add Hook) button. This runs the `gatomia.hooks.addHook` command and opens the hook editor form.
  </Step>
  <Step title="Configure the trigger">
    Set the conditions that determine when the hook fires:

    - **Agent**: Select `speckit` or `openspec`
    - **Operation**: Choose the operation to watch (e.g., `plan`, `specify`, `clarify`)
    - **Timing**: Select `after` (default) or `before` agent execution

    Agent command format for validation: commands must start with `/speckit.` or `/openspec.`.
  </Step>
  <Step title="Choose an action type">
    Select one of the five action types:

    | Type | What it does |
    |---|---|
    | **MCP** | Calls one or more Model Context Protocol tools |
    | **Agent** | Runs a built-in SpecKit or OpenSpec command |
    | **Git** | Executes a `commit` or `push` operation |
    | **GitHub** | Creates or updates GitHub issues or pull requests |
    | **Custom** | Invokes a custom agent from your agent registry |
  </Step>
  <Step title="Configure the action (MCP example)">
    For MCP actions, click the **MCP tool picker** to browse available servers and tools discovered from your GitHub Copilot configuration. Select one or more tools, write an instruction prompt, and map any parameters using template variables or literal values.
  </Step>
  <Step title="Save the hook">
    Click **Save**. The hook appears in the Hooks view and is enabled by default.
  </Step>
</Steps>

## Enable, disable, and monitor hooks

Use the Hooks view to manage the lifecycle of existing hooks:

- **Toggle on/off**: Click the play icon to enable or the pause icon (`gatomia.hooks.enable` / `gatomia.hooks.disable`) to disable a hook without deleting it.
- **View execution logs**: Click the **eye** icon (View Hook Logs) to open the execution log panel. Logs show the trigger timestamp, status, duration, chain depth, and any error details.
- **Export / import**: Use the toolbar export (`gatomia.hooks.export`) and import (`gatomia.hooks.import`) buttons to save your hook configurations as JSON or load them from another machine.

## Template variables

Use these variables inside action prompts and message templates to inject context at execution time.

| Variable | Description |
|---|---|
| `$agentOutput` | File content generated by the triggering agent |
| `$clipboardContent` | Clipboard content at the moment of execution |
| `$outputPath` | Path to the file generated by the agent |
| `$specId` | Unique identifier of the current spec |
| `$specTitle` | Title of the current spec |
| `$specStatus` | Current lifecycle status of the spec |
| `$timestamp` | ISO timestamp of the trigger event |
| `$branch` | Current git branch name |
| `$user` | Name of the user who triggered the event |

Reference variables with the `$` prefix in any prompt or message template field. For Git commit message templates, use curly-brace syntax: `{branch}`, `{feature}`, `{user}`.

## Control timing

<AccordionGroup>
  <Accordion title="After execution (default)">
    The hook runs after the agent or operation completes. Use this for notifications, artifact processing, and CI/CD triggers. The `$agentOutput` and `$outputPath` variables are available because the agent has already written its files.
  </Accordion>
  <Accordion title="Before execution">
    The hook runs before the agent starts. Use this for pre-flight validation or environment setup. Enable **Wait for completion** to block the agent from proceeding until the hook finishes — useful when the hook must prepare something the agent depends on.
  </Accordion>
</AccordionGroup>

<Note>
  GatomIA implements graceful degradation for MCP hooks. If an MCP server is unavailable when a hook fires, the hook fails with a logged error but does not crash the extension or block the triggering agent. GatomIA also uses automatic retry logic with exponential backoff for transient MCP server failures, so temporary network issues resolve themselves without manual intervention.
</Note>
