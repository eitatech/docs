---
title: "Automate dev workflows with GatomIA Hooks and MCP triggers"
description: "Learn how GatomIA Hooks let you trigger MCP tools, Copilot agents, Git operations, and GitHub API calls automatically when development events occur."
---

**Hooks** are GatomIA's event-driven automation system—think of them as "IFTTT for developers." You define a rule that says "when this event happens, run that action," and GatomIA handles the rest without any manual intervention. Hooks let you automate repetitive tasks such as creating GitHub issues when a spec moves to review, running a code analysis agent after every commit, or committing approved specs to Git automatically. Open the **Hooks** view in the Activity Bar to create, enable, disable, and monitor your hooks.

## Trigger sources

A hook fires when a matching event occurs in your workspace. GatomIA supports four categories of triggers:

| Trigger type | Examples |
|--------------|---------|
| **Spec status change** | Spec moves from `draft` → `ready-for-review`, or from `in-review` → `approved` |
| **Git operation** | Commit created, pull request opened, branch pushed |
| **File change** | A specific file or pattern is saved or deleted |
| **Manual trigger** | You invoke the hook directly from the Hooks view |

## Action types

When a trigger fires, GatomIA executes one of five action types:

<AccordionGroup>
  <Accordion title="MCP tool execution">
    Call any tool exposed by an MCP (Model Context Protocol) server configured in your GitHub Copilot setup. Browse available servers and tools from the Hooks view's MCP browser, map parameters using template variables or literal values, and GatomIA calls the tool automatically. Examples: create a Jira ticket, send a Slack notification, post a GitHub comment.
  </Accordion>
  <Accordion title="GitHub Copilot agent">
    Run a built-in GitHub Copilot agent command such as `/speckit.clarify` or `/speckit.refine`. The agent executes with the hook's context already populated—no manual copy-pasting required.
  </Accordion>
  <Accordion title="Git operations">
    Perform Git operations with templated commit messages. Supported operations include `commit`, `push`, and `pull`. You can configure the hook to push to the remote automatically after committing.
  </Accordion>
  <Accordion title="GitHub API">
    Interact with the GitHub REST API to create issues, open pull requests, or close existing issues. Use template variables in the title and body to include dynamic context such as the spec name or author.
  </Accordion>
  <Accordion title="Custom agents">
    Invoke any agent registered in GatomIA's agent registry—including your own `.agent.md` agents—using the `source:name` format (e.g., `local:code-reviewer`). Pass a prompt and optional MCP tools alongside the invocation.
  </Accordion>
</AccordionGroup>

## Execution timing

Every hook has a **timing** setting that controls when it runs relative to an agent operation:

- **After execution** (default): The hook runs after the triggering agent or operation completes. Use this for notifications, artifact processing, and CI/CD triggers.
- **Before execution**: The hook runs before the agent starts. Enable the **wait for completion** option to block the agent from proceeding until the hook finishes—useful for pre-flight validation or environment setup.

## Template variables

Use template variables in hook action parameters to inject dynamic context at runtime. Variables follow the `$variableName` format (shown in the variable picker inside the hook editor):

| Variable | Available for | Description |
|----------|--------------|-------------|
| `$timestamp` | All triggers | ISO timestamp when the hook fired |
| `$branch` | Git triggers | Current Git branch name |
| `$user` | All triggers | User who triggered the event |
| `$specId` | Spec triggers | Unique spec identifier |
| `$specTitle` | Spec triggers | Human-readable spec title |
| `$specStatus` | Spec triggers | Current spec status |
| `$agentOutput` | After-execution hooks | File content generated by the triggering agent |
| `$clipboardContent` | After-execution hooks | Clipboard content at the time of execution |
| `$outputPath` | After-execution hooks | File system path to the agent-generated file |

<Note>
  Output capture variables (`$agentOutput`, `$clipboardContent`, `$outputPath`) are only available in **after-execution** hooks. They resolve to empty strings if no output was captured.
</Note>

## Hook storage

Hooks are stored in your workspace at `.vscode/gatomia/hooks.json` as a JSON array. GatomIA manages this file for you through the Hooks view UI, but you can also import and export hook configurations to share them with your team.

## Creating a hook

<Steps>
  <Step title="Open the Hooks view">
    Click the GatomIA icon in the Activity Bar and select the **Hooks** view.
  </Step>
  <Step title="Click Create New Hook">
    Give the hook a name (up to 100 characters) and choose a trigger type and its conditions.
  </Step>
  <Step title="Select an action type">
    Choose from MCP, Agent, Git, GitHub, or Custom. For MCP actions, use the MCP browser to select a server and tool, then map parameters using the variable picker or literal values.
  </Step>
  <Step title="Set timing">
    Choose whether the hook runs before or after the triggering operation. Enable **wait for completion** if you need the hook to block forward progress.
  </Step>
  <Step title="Save and enable">
    Save the hook. It is enabled by default. Toggle the switch in the Hooks view to disable it without deleting it.
  </Step>
</Steps>

## Monitoring hooks

The Hooks view shows each hook's enabled state, last execution time, and total execution count. Click a hook to view its execution log, which includes:

- The trigger event and context snapshot
- The action that was executed and its parameters
- Execution duration and success or failure status
- Error details for failed executions

GatomIA retries transient failures automatically and degrades gracefully when MCP servers are temporarily unavailable—failed hooks are logged but do not crash the extension or block your workflow.
